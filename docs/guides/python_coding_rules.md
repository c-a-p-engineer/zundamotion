# Python Coding Rules

Zundamotion の Python コードを、人間と AI / Codex の両方が低コストで読める状態に保つための実装規約。
このファイルは Python 実装を追加、修正、分割するときに参照する。

## 1. 目的

- 1 回の変更で読む範囲を狭くし、不要な入出力トークンを減らす
- ファイル、関数、クラスの責務を明確にし、肥大化を防ぐ
- 命名と構造を揃えて、関連コードを推測しやすくする
- 副作用と純粋処理を分離し、テストしやすくする

## 2. 最初に読む順番

1. `AGENTS.md`
2. `docs/guides/ai_coding_rules.md`
3. `docs/guides/project_structure.md`
4. このファイル
5. 対象モジュール

性能変更なら `docs/guides/performance_regression_ledger.md`、CLI 変更なら `docs/guides/setup_and_runtime.md` も先に確認する。

## 3. フォルダ構成

### ルート直下

- `zundamotion/`
  - 本体コードを置く。アプリケーションロジックは原則ここへ集約する
- `tools/`
  - 開発補助や運用補助のスクリプトを置く。ライブラリ本体の責務は持たせない
- `scripts/`
  - YAML やサンプル入力を置く。実装ロジックを持ち込まない
- `docs/`
  - 仕様、設計、運用、判断記録を置く。長い説明をコードコメントへ重複させない

### `zundamotion/` 配下

- `components/<domain>/`
  - 音声、設定、字幕、動画など、ドメイン単位の実装を置く
- `components/pipeline_phases/`
  - pipeline の各フェーズを置く。順序制御本体と細部実装を分ける
- `utils/`
  - 複数ドメインから共有される低レベル処理だけを置く
- `reporting/`
  - 集計、統計、レポート用途の処理を置く

### 配置ルール

- 新規コードは「どの入力を受け、どの出力を返すか」が近い場所へ置く
- `utils/` は最後の手段とし、ドメイン固有処理を安易に集めない
- CLI 専用処理は `main.py` 周辺へ寄せ、内部ロジックへ CLI 事情を漏らさない
- YAML ロード、環境変数解決、FFmpeg 実行、VOICEVOX 呼び出しは境界として分離する

## 4. ファイルサイズと分割基準

- 1 ファイルは目安 200〜400 行、上限 500 行
- 1 クラスは目安 150 行以内、200 行を超えたら責務分割を検討する
- 1 関数は目安 20〜40 行、上限 80 行
- public 関数は 7 個前後を超えたら別モジュール分割を検討する
- import を除いた実ロジックが 300 行を超えるなら、先に分割可能性を確認する

次のいずれかに当てはまるなら分割する。

- 入出力、変換、検証、実行が 1 ファイルに混在している
- 同じ関数でパス解決、設定解決、外部 I/O、結果整形を全部やっている
- 一部だけを読みたいのに、前提として長い周辺コードまで追う必要がある
- 修正時に毎回スクロールして別責務の塊をまたぐ

## 5. 複雑性の粒度

- 1 関数は 1 つの判断単位に絞る
- 分岐は 3 段以上ネストさせない。深くなる場合は早期 return か補助関数へ逃がす
- 1 関数で扱う主要データ構造は 2〜3 種までを目安にする
- ループの中でさらに複雑な分岐や外部 I/O を重ねない
- 条件分岐が 5 個以上並ぶ場合は、判定表、設定オブジェクト、専用関数への分離を検討する
- 例外処理は失敗箇所に近い層で文脈を付け、握りつぶさない

複雑性を下げる優先順:

1. 早期 return を使う
2. 純粋変換を補助関数へ切り出す
3. 境界 I/O を呼び出し元へ寄せる
4. データ構造を小さな型へ置き換える

## 6. 命名規約

- ファイル名: `snake_case.py`
- 関数名: `snake_case`
- 変数名: `snake_case`
- クラス名: `PascalCase`
- 定数名: `UPPER_SNAKE_CASE`
- private 補助関数・属性: `_leading_underscore`
- bool 値: `is_`, `has_`, `should_`, `can_` で始める

### 命名の方針

- 名前だけで入出力が推測できる動詞を使う
- `data`, `info`, `temp`, `util`, `helper`, `misc` のような曖昧名を避ける
- `build_` は構築、`load_` は読み込み、`resolve_` は解決、`render_` は生成、`run_` は実行に限定する
- `validate_` は検証のみ、補正や保存を含めない
- FFmpeg ラベルや一時変数にも意味が分かる接頭辞を付ける

### 略語

- 一般的な略語以外は避ける
- `cfg`, `ctx`, `msg` のような短縮は、近いスコープで意味が明確な場合だけ許容する
- ドメイン用語は README や docs と同じ表記に揃える

## 7. 型とデータ境界

- public 関数は引数と戻り値の型を明記する
- `Dict[str, Any]` は YAML 境界、JSON 境界、外部 I/O 境界でのみ許容する
- 内部ロジックへ入る前に、小さな型、dataclass、既存モデルへ変換する
- `Optional` を返す関数は、`None` の意味を docstring か命名で明確にする
- 返り値が 3 項目以上のタプルになる場合は、専用型を検討する

## 8. 副作用と I/O

- ファイル I/O、ネットワーク I/O、subprocess 実行、環境変数読み取りは局所化する
- 純粋変換関数は、ログ出力や外部状態更新を持ち込まない
- FFmpeg コマンド生成と実行を同じ関数へ混ぜない
- VOICEVOX 呼び出し、probe、キャッシュ更新は境界関数で扱う
- 環境変数は起動時や設定解決時にまとめて読み、深い層で再読しない

## 9. コメントと docstring

- ファイル冒頭に、責務、入口、関連ファイルを最大 10 行で書く
- コメントは「何をしているか」ではなく「なぜ必要か」を補う
- 長い仕様説明や運用判断は docs に置き、コードへ重複転記しない
- public API と複雑な変換関数には短い docstring を付ける
- TODO は期限か判断条件なしに増やさず、必要なら `docs/issues_pending.md` へ残す

## 10. import と依存

- 標準ライブラリ、サードパーティ、ローカルモジュールの順でまとめる
- 未使用 import を残さない
- 循環 import を避けるため、責務分割を先に見直す
- 新規依存追加時は、必要性、保守性、サイズ、ライセンス、既存依存で代替できない理由を確認する

## 11. ログとエラー

- `print` ではなく既存 logger を使う
- エラーは入力不備、外部 I/O、内部バグのどれかを判別しやすくする
- ログメッセージは検索しやすい語彙で書く
- DEBUG ログだけ見れば、外部コマンドや主要パラメータを再現できる状態を目指す
- 例外を catch したら、原因調査に必要な文脈を追加して再送出する

## 12. テストしやすさ

- 純粋変換は副作用から切り離し、単体で検証可能にする
- 時刻、乱数、環境変数、パス、外部コマンドは注入しやすくする
- 1 つのテストで複数責務を同時に検証しない
- テストのためだけに本体コードへ不自然な分岐を足さない

## 13. AI / Codex 向け追加ルール

- 変更前に、対象ファイルとその直接依存だけを先に読む
- 1 回の変更で読むファイルは、原則 3〜6 個以内に収める
- 500 行近いファイルへ追記する前に、別ファイルへ逃がせないか検討する
- 1 つの関数へ複数責務を足して短期的に済ませない
- 同じ仕様を README、docs、コメントへ 3 重記載しない
- PR や作業報告では、読んだファイルと読まなかった理由を短く書ける構造を維持する

## 14. 変更前チェック

- 置き場所は `components/<domain>/`、`pipeline_phases/`、`utils/` のどこが妥当か
- 既存関数へ足すより新規関数へ逃がした方が責務が明確か
- 名前だけで責務が推測できるか
- 型境界は外側で閉じているか
- docs へ移すべき長文説明をコードへ書こうとしていないか
